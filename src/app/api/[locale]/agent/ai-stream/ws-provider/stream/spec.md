# AI Stream Protocol — WS-Based Provider

## What This Is

A WebSocket-native protocol for AI inference. The server runs the AI loop and pushes all events to connected clients. Tool execution is bidirectional: the server dispatches requests to the client, the client executes locally and broadcasts results back.

HTTP streaming APIs are a compatibility layer on top. The WS stream is the native interface. Any LLM provider that implements this protocol can be used interchangeably — including on-prem models, commercial APIs, and relay nodes like unbottled.ai.

---

## Roles

**Provider** — runs the AI inference loop. Generates all IDs (thread, message, tool call). Emits all stream events. Requests tool execution from the client.

**Client** — owns the user session, credentials, and tool implementations. Subscribes to the provider's event channel. Executes tools locally. Broadcasts results back to the provider. Mirrors the full message tree using provider-assigned IDs verbatim.

---

## ID Contract

All IDs are generated server-side at stream initiation:

- **threadId** — generated by the provider if not supplied by the client (continuation). Stable across sessions.
- **userMessageId** — generated by the provider when the session starts.
- **aiMessageId** — pre-allocated before the AI loop starts; arrives in the first `message-created` event.
- **callId** — generated per tool invocation at dispatch time. Scoped to the session.

Clients store all IDs verbatim. No translation layer. Cross-platform sync is a direct ID comparison.

## Session Lifecycle

### 1. Initiate

Client sends HTTP POST to the provider:

```
content         string           — user message text (may be empty for answer-as-ai)
model           string           — model identifier (ChatModelId)
skill?          string           — system prompt / persona ID (default: "default")
threadId?       string           — continue an existing thread (omit for new)
systemPrompt?   string           — additional system instructions injected before skill
tools?          ToolSpec[]       — tools the AI may call (see Tool Advertisement)
timezone?       string           — IANA timezone (default: "UTC")
instanceId?     string           — caller instance ID (for subfolder routing)
```

Provider responds synchronously (stream runs fire-and-forget in background):

```
responseThreadId   string   — thread ID (use for WS channel subscription)
messageId          string   — pre-allocated ID for the first assistant message
```

### 2. Subscribe

Client opens WebSocket to provider and sends:

```json
{ "type": "subscribe", "channel": "agent/chat/threads/{threadId}/messages" }
```

**The subscription MUST be established before the AI loop emits its first event.** Open the WS before or immediately after the HTTP POST — implementations that open it concurrently risk losing early events.

### 3. Stream and Tool Loop

Provider runs the AI loop. Events flow on the channel. When the AI calls a tool:

1. Provider emits `tool-execute-request` with `{ callId, toolName, args }`.
2. Client executes the tool locally.
3. Client POSTs `tool-execute-result` to the provider's `/ws/broadcast`:
   ```
   POST /ws/broadcast
   { channel, event: "tool-execute-result", data: { callId, result, error? } }
   ```
4. Provider's WS subscription on the same channel receives the event.
5. Provider resolves the pending promise keyed by `callId`. AI loop continues.

Parallel tool calls each have their own `callId` and may resolve in any order. Timeout per unresolved call: 120 seconds.

### 4. Terminate

Provider emits `stream-finished`. Client drains its event queue (all async DB writes must complete — `credits-deducted` is guaranteed before `stream-finished`), closes the socket.

## Wire Format

All messages share a common envelope:

```
channel   string    — "agent/chat/threads/{threadId}/messages"
event     string    — event type key
data      object    — event payload
seq       integer   — global monotonic counter (enables replay/resumability)
```

### Events: Provider → Client

| Event                  | Payload                                                                                                                                                              |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `message-created`      | `{ messages: [{ id, threadId, role, isAI, content, parentId, sequenceId, model, skill, metadata }], streamingState }`                                                |
| `content-delta`        | `{ messages: [{ id, content: delta }] }` — append to existing content                                                                                                |
| `content-done`         | `{ messages: [{ id, content, metadata: { totalTokens, finishReason } }] }`                                                                                           |
| `reasoning-delta`      | `{ messages: [{ id, content: delta }] }` — reasoning models only, append                                                                                             |
| `reasoning-done`       | `{ messages: [{ id, content }] }` — reasoning models only                                                                                                            |
| `tool-execute-request` | `{ callId, toolName, args }` — client must execute and respond                                                                                                       |
| `tool-result`          | `{ messages: [{ id, metadata: { toolCall } }] }` — provider applied result                                                                                           |
| `tokens-updated`       | `{ messages: [{ id, metadata: { promptTokens, completionTokens, totalTokens, cachedInputTokens, cacheWriteTokens, timeToFirstToken, finishReason, creditCost } }] }` |
| `credits-deducted`     | `{ amount }` — always arrives before `stream-finished`                                                                                                               |
| `files-uploaded`       | `{ messages: [{ id, metadata: { isUploadingAttachments: false, attachments: [...] } }] }` — file upload complete                                                     |
| `voice-transcribed`    | `{ messages: [{ id, content: transcribedText, metadata: { isTranscribing: false } }] }` — STT complete                                                               |
| `audio-chunk`          | `{ audioData, chunkIndex, audioMessageId, audioIsFinal, audioText }` — TTS output                                                                                    |
| `stream-finished`      | `{ streamingState, reason }` — reason: `completed \| cancelled \| error \| timeout`                                                                                  |
| `error`                | `{ message, errorType }` — persisted as an error message on the client                                                                                               |

### Events: Client → Provider (tool results only)

```
POST /ws/broadcast
{ channel, event: "tool-execute-result", data: { callId, result, error? } }
```

No other client-initiated messages after subscribe. All loop control belongs to the provider.

---

## User Isolation

Events are delivered only to the authenticated user who initiated the session. Identity is `userId` for authenticated users, `leadId` for anonymous/public sessions. Authentication via httpOnly cookies (browser) or `?token=&leadId=` URL params (server-to-server relay).

---

## Thread Modes

**Normal** — provider stores conversation. Client passes `threadId` to continue. Provider thread is authoritative.

**Incognito** — no `threadId` passed; provider creates an ephemeral thread. No messages persisted on provider. Client mirrors events to local storage.

---

## Multi-Turn Message Tree

Every message has a `parentId`. The client reconstructs the tree in `seq` order:

```
user message          (id: A, parentId: null)
  └── assistant       (id: B, parentId: A) — turn 1
        ├── tool      (id: C, parentId: B) — tool call
        │     └── assistant  (id: D, parentId: C) — turn 2 after tool result
        └── [deferred tool result may also land here as a sibling]
```

### Branching

The provider tracks `leafMessageId` — the current branch tip. New messages always append to the current leaf. When a user sends a new message from a different branch point (e.g., retrying from an earlier message), the provider creates a new branch and subsequent messages follow that path.

**Operations that affect branching:**

| Operation       | What happens                                                                                                    |
| --------------- | --------------------------------------------------------------------------------------------------------------- |
| `send`          | New user message appended to current leaf                                                                       |
| `retry`         | New assistant message appended to the same parent as the previous attempt                                       |
| `edit`          | User message re-parented; everything below the edit point is orphaned from the active chain                     |
| `answer-as-ai`  | New assistant message appended with no new user message                                                         |
| `wakeup-resume` | Revival appends deferred tool result and new assistant turn to the stored `leafMessageId` at task creation time |

Branch tracking for deferred results: every async task stores `leafMessageId` at dispatch time. On completion, the result appends to that stored leaf — even if the user switched branches while the task ran.

## File Attachments

Client passes files as part of the initiation request:

```
attachments   File[]   — images, PDFs, audio, video
audioInput    { file: File | null }   — voice input (STT)
```

### Attachment Lifecycle

1. **Upload in progress**: `message-created` for the user message includes `metadata.isUploadingAttachments: true`.
2. **Upload complete**: `files-uploaded` event updates the message with `metadata.isUploadingAttachments: false` and the full attachment array.
3. **Context injection**: Provider includes attachment URLs in the AI's context. For models that don't support a given modality, the attachment is stripped and the AI is warned.

### Attachment Metadata Shape

Each processed attachment:

```
id          string   — unique file ID
url         string   — storage URL (signed or permanent)
filename    string   — original filename
mimeType    string   — e.g. "image/png", "application/pdf", "audio/webm"
size        number   — bytes
data?       string   — optional base64 (reliability fallback)
```

### Modality Handling

- **Images**: passed as vision parts to multimodal models; stripped with a warning for text-only models.
- **Audio**: transcribed via STT before the AI runs (triggers `voice-transcribed` event); transcription text replaces audio in context.
- **PDFs**: extracted as text and injected into context.
- **Video**: treated as audio+image; extracted frames and/or audio passed to model.

---

## Voice Mode

When `voiceMode.enabled = true`:

1. Audio input (`audioInput.file`) is transcribed via STT before the AI runs.
2. Transcription emitted as `voice-transcribed` event updating the user message content.
3. AI runs against the transcribed text (not the raw audio).
4. AI response text is synthesized via TTS, emitted in chunks as `audio-chunk` events.
5. `audioIsFinal: true` on the last chunk signals playback can start.

---

## Deduplication

Clients MUST deduplicate `message-created` events by `id`. The provider may emit the same event more than once in edge cases (reconnect replay, parallel channel subscribers).

---

## Caching Properties

All inference runs on the provider. The provider's KV cache is shared across all clients:

- System prompts and skill instructions cache once, hit on every request using the same model + skill.
- Conversation prefix accumulates as cached context — per-turn cost falls as threads grow.
- Tool definitions (static per skill) cache after the first turn.

Clients running local inference for a single user get no cross-session cache benefit. A provider serving many clients with the same model hits the cache on nearly every request.

---

## Tool Advertisement

Client declares tools in the initiation request:

```typescript
tools: [{
  name: string          — unique tool name (e.g. "web_search")
  description: string   — shown to AI as tool description
  parameters: object    — JSON Schema object describing the input
}]
```

The AI receives these tool definitions alongside any server-side tools. When the AI calls a client-declared tool, the provider emits `tool-execute-request` and waits for `tool-execute-result` via broadcast.

Server-side tools (registered endpoints) are always available unless filtered by the caller. Client-declared tools overlay the server-side set — same name = override.

---

## Self-Relay

When the provider URL resolves to the local instance, no HTTP+WS round-trip is needed. The stream runs in-process, bypassing the network entirely. Used for self-hosted setups where the local node IS the provider, and for testing.

---

## Provider Compliance

A compliant provider implements:

1. HTTP POST endpoint → returns `{ responseThreadId, messageId }` synchronously
2. WebSocket endpoint accepting `{ type: "subscribe", channel }` messages
3. Full event set emitted on the subscribed channel (including `files-uploaded` and `voice-transcribed` when applicable)
4. `/ws/broadcast` POST endpoint — accepts `{ channel, event, data }` and distributes to subscribers
5. Server-generated IDs for all entities
6. Branch tracking: `parentId` on every `message-created`, correct `leafMessageId` threading for deferred results
7. File attachment upload → `files-uploaded` event lifecycle
8. Modality filtering: strip unsupported attachment types, warn AI when stripped

Legacy HTTP streaming providers can adapt: map their SSE event stream to this event format, expose `/ws/broadcast` for tool results, use the same ID namespace. The WS subscription is the native path; SSE is a compatibility shim.
