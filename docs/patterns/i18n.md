# Scoped Translation System

> **Part of NextVibe Framework** (GPL-3.0) - Module-specific type-safe internationalization

Each module has its own isolated translation scope with SHORT, relative keys. No more global `"app.api.*"` prefixes.

---

## ⛔ ABSOLUTE PROHIBITIONS - Read This First

These patterns are **NEVER acceptable**:

```typescript
// ❌❌❌ NEVER call scopedT() with a hardcoded or default locale - EVER
const { t } = scopedTranslation.scopedT("en-GLOBAL"); // FORBIDDEN: hardcoded locale
const { t } = scopedTranslation.scopedT(defaultLocale); // FORBIDDEN: default locale shortcut

// ❌ route creating t itself instead of using the handler prop
handler: async (props) => {
  const { t } = scopedTranslation.scopedT(props.locale); // ← WRONG: t is already in props!
};

// ❌ definition importing parent i18n instead of module-local
import { scopedTranslation } from "../../i18n"; // wrong scope - use ./i18n
```

---

## ✅ The Two Correct Patterns: `t` vs `locale`

The rule is simple: **does this module get called from outside its own `route.ts`?**

### Pattern A - Internal module (called only from its own `route.ts`)

`t` is created once by `endpointsHandler` and flows as a prop. Receive it directly.

```typescript
// ✅ repository.ts - receive t as parameter
import type { scopedTranslation } from "./i18n"; // type-only import
type ModuleT = ReturnType<typeof scopedTranslation.scopedT>["t"];

export async function createSmtpAccount(params: {
  data: CreateInput; logger: EndpointLogger; t: ModuleT;
}): Promise<ResponseType<SmtpAccount>> {
  const { data, logger, t } = params;
  // use t() directly - never call scopedT() here
}

// ✅ route.ts - destructure t from handler props, pass it through
handler: async ({ data, logger, t }) => {
  return SmtpRepository.createSmtpAccount({ data, logger, t });
},
```

### Pattern B - External module (called from outside its own `route.ts`)

When a module is called from native repos, ai-stream, story pages, cron jobs, or any other server function outside its `route.ts`, it must accept `locale` and create `t` internally.

```typescript
// ✅ repository.ts - accept locale, create t internally
import { scopedTranslation } from "./i18n"; // value import (not type-only)
import type { CountryLanguage } from "@/i18n/core/config";

export class ThreadsRepository {
  static async ensureThread(
    params: { threadId?: string; locale: CountryLanguage; logger: EndpointLogger; ... }
  ): Promise<...> {
    const { locale, logger } = params;
    const { t } = scopedTranslation.scopedT(locale); // ✅ OK: called from ai-stream/headless
    // ...
  }
}

// ✅ native repository.ts - must accept locale to match server interface
static async getFolder(
  user: JwtPayloadType,
  data: GetInput,
  logger: EndpointLogger,
  locale: CountryLanguage, // ← matches server signature
): Promise<ResponseType<FolderOutput>> {
  // native doesn't use t, but locale param keeps the interface consistent
}
```

**Decision rule:**

- Module only called from `endpointsHandler` in its own `route.ts` → **accept `t`** (cleaner, no re-creation)
- Module called from anywhere else (native repos, ai-stream, cron, story pages, other repos) → **accept `locale: CountryLanguage`** and create `t` internally with `scopedTranslation.scopedT(locale)`

**Why locale for external modules**: The `endpointsHandler` infrastructure creates `t` from the HTTP request. External callers (native repos, server functions) don't go through `endpointsHandler`, so they pass `locale` directly. Accepting `locale` also keeps native and server interfaces consistent for compile-time type checking.

## Scoping Strategy - How Many i18n Scopes?

The goal is **one scope per logical domain**, not one scope per file or per endpoint. Use judgment based on the size and cohesion of the area.

### Small domain → one shared scope

A small, cohesive area (a handful of endpoints, one conceptual thing) should share a single i18n scope at the domain root. All definitions in that domain import from the same `./i18n` (which is the domain root's `i18n`).

**Example: `subscription/`** - a few endpoints, one concept → single scope at `subscription/i18n/`

```
src/app/api/[locale]/subscription/
├── i18n/               ← one shared scope for the whole domain
│   ├── index.ts
│   ├── en/index.ts
│   ├── de/index.ts
│   └── pl/index.ts
├── [tabId]/definition.ts     ← imports from ../i18n
├── components/...
└── route.ts
```

### Large domain → multiple sub-scopes

A large domain with many distinct sub-features (each with their own vocabulary, forms, and error messages) should have a scope per logical sub-area. The parent scope aggregates sub-scope translations for breadcrumb labels and shared strings.

**Example: `emails/`** - IMAP client, SMTP client, templates, journeys → separate scopes per sub-feature

```
src/app/api/[locale]/emails/
├── i18n/                       ← parent scope (category label, shared strings)
│   ├── en/index.ts             ← imports sub-scope translations, nests them
│   └── ...
├── imap-client/
│   ├── i18n/                   ← sub-scope for IMAP client
│   └── ...
├── smtp-client/
│   ├── i18n/                   ← sub-scope for SMTP client
│   └── ...
└── templates/
    ├── i18n/                   ← sub-scope for templates
    └── ...
```

### Decision guide

| Signal                                                | Recommendation                 |
| ----------------------------------------------------- | ------------------------------ |
| < ~5 endpoints, one concept                           | Single scope at domain root    |
| Many endpoints with distinct vocabularies             | Separate scope per sub-feature |
| Endpoints share the same form fields / error messages | Keep in one scope              |
| Sub-feature has its own enum set                      | Give it its own scope          |

**Don't** create a new scope for every single endpoint or route file. That creates fragmentation and makes parent aggregation noisy. **Do** group by logical sub-feature boundaries.

## File Structure

```
src/app/api/[locale]/{module}/
├── i18n/
│   ├── index.ts        ← createScopedTranslation() + export TranslationKey type
│   ├── en/index.ts     ← English translations (source of truth, no type annotation)
│   ├── de/index.ts     ← German (typed: typeof enTranslations)
│   └── pl/index.ts     ← Polish (typed: typeof enTranslations)
├── definition.ts       ← uses scopedTranslation from ./i18n (ALWAYS module-local)
├── enum.ts             ← createEnumOptions<ModuleTranslationKey>(...)
└── repository.ts       ← internal: receives t; external: receives locale and creates t internally
```

**Rules:**

1. One `index.ts` per language folder - NO other files, NO subdirectories
2. English file has NO type annotation, NO `as const`
3. German/Polish MUST have `: typeof enTranslations` annotation
4. Keys are SHORT and relative: `"create.title"` not `"emails.smtpClient.create.title"`
5. Parents import only direct children (not grandchildren)

## Implementation

### i18n/index.ts

```typescript
import { createScopedTranslation } from "@/i18n/core/scoped-translation";
import { translations as deTranslations } from "./de";
import { translations as enTranslations } from "./en";
import { translations as plTranslations } from "./pl";

export const scopedTranslation = createScopedTranslation({
  en: enTranslations,
  de: deTranslations,
  pl: plTranslations,
});

export type SmtpClientTranslationKey =
  (typeof scopedTranslation)["ScopedTranslationKey"];
```

### i18n/en/index.ts (Source of Truth)

```typescript
export const translations = {
  category: "SMTP Client",
  create: {
    title: "Create SMTP Account",

  },
  enums: {

```

## Usage by File Type

### definition.ts

Pass `scopedTranslation` to `createEndpoint()` and use `scoped*` field factory functions for type-safe scoped keys. Use the **new flat API** (`objectField`, `responseArrayField`) - config is one object containing `usage` + `children`/`child`:

**Important:** Always import from `./i18n` (module-local), never from a parent like `../../i18n`. The `t` type in `route.ts` is inferred from the endpoint's `scopedTranslation` - if definition and repository use different scopes, types won't align.

```typescript
import { scopedTranslation } from "./i18n"; // ← always module-local, NOT ../../i18n
import {
  objectField,
  responseArrayField,
  requestField,
  responseField,
  scopedBackButton,
  scopedSubmitButton,
} from "@/app/api/[locale]/system/unified-interface/shared/field/utils-new";

export const { POST } = createEndpoint({
  scopedTranslation, // ← required: enables scoped key validation
  title: "create.title",
  description: "create.description",
  fields: objectField(scopedTranslation, {
    type: WidgetType.CONTAINER,
    usage: { request: "data", response: true },
    children: {
      backButton: scopedBackButton(scopedTranslation, {
        usage: { request: "data", response: true },
      }),
      email: requestField(scopedTranslation, {
        type: WidgetType.FORM_FIELD,
        fieldType: FieldDataType.EMAIL,
        label: "create.form.email.label",
        schema: z.string().email(),
      }),
      items: responseArrayField(scopedTranslation, {
        type: WidgetType.CONTAINER,
        child: objectField(scopedTranslation, {
          type: WidgetType.CONTAINER,
          usage: { response: true },
          children: {
            name: responseField(scopedTranslation, {
              type: WidgetType.TEXT,
              schema: z.string(),
            }),
          },
        }),
      }),
    },
  }),
  // ...
});
```

**All scoped field functions** take `scopedTranslation` as first param (type inference only). Import from `utils-new`:

```typescript
import {
  // Primitive fields
  requestField,
  responseField,
  requestResponseField,
  requestUrlPathParamsField,
  requestUrlPathParamsResponseField,
  // Object fields
  objectField,
  // Array fields
  responseArrayField,
  requestDataArrayField,
  responseArrayOptionalField,
  // Widget / button fields
  widgetField,
  scopedBackButton,
  scopedNavigateButtonField,
  scopedEditButton,
  scopedDeleteButton,
  scopedSubmitButton,
  // Custom widget (no st needed)
  customWidgetObject,
} from "@/app/api/[locale]/system/unified-interface/shared/field/utils-new";
```

**Flat API** - config is a single object containing everything:

```typescript
// Object with usage + children in one config
fields: objectField(scopedTranslation, {
  type: WidgetType.CONTAINER,
  usage: { request: "data" },
  children: {
    email: requestField(scopedTranslation, { ... }),
  },
}),

// Array with child in one config
items: responseArrayField(scopedTranslation, {
  type: WidgetType.CONTAINER,
  child: objectField(scopedTranslation, {
    type: WidgetType.CONTAINER,
    usage: { response: true },
    children: { name: responseField(scopedTranslation, { ... }) },
  }),
}),
```

> **NOTE:** All field functions now use the flat single-config API. Import from `utils-new`, not `utils`.

Non-scoped equivalents (internal reference - prefer scoped versions):

| Non-scoped (internal)                       | Scoped (canonical - use these)                  |
| ------------------------------------------- | ----------------------------------------------- |
| `objectFieldNew(config)`                    | `objectField(st, config)`                       |
| `responseArrayFieldNew(config)`             | `responseArrayField(st, config)`                |
| `requestDataArrayFieldNew(config)`          | `requestDataArrayField(st, config)`             |
| `responseArrayOptionalFieldNew(config)`     | `responseArrayOptionalField(st, config)`        |
| `requestField(config)`                      | `requestField(st, config)`                      |
| `responseField(config)`                     | `responseField(st, config)`                     |
| `requestResponseField(config)`              | `requestResponseField(st, config)`              |
| `requestUrlPathParamsField(config)`         | `requestUrlPathParamsField(st, config)`         |
| `requestUrlPathParamsResponseField(config)` | `requestUrlPathParamsResponseField(st, config)` |
| `widgetField(ui)`                           | `widgetField(st, ui)`                           |
| `backButtonNew(config)`                     | `scopedBackButton(st, config)`                  |
| `navigateButtonFieldNew(config)`            | `scopedNavigateButtonField(st, config)`         |
| `editButtonNew(config)`                     | `scopedEditButton(st, config)`                  |
| `deleteButtonNew(config)`                   | `scopedDeleteButton(st, config)`                |
| `submitButtonNew(config)`                   | `scopedSubmitButton(st, config)`                |

### enum.ts

Pass `scopedTranslation` as first argument - this validates enum label strings against the module's schema at compile time:

```typescript
import { createEnumOptions } from "@/app/api/[locale]/system/unified-interface/shared/field/enum";
import { scopedTranslation } from "./i18n"; // value import

export const { enum: SmtpSecurityType } = createEnumOptions(scopedTranslation, {
  NONE: "enums.smtpSecurityType.none",
  TLS: "enums.smtpSecurityType.tls",
});
```

### repository.ts

**Every `fail()` MUST use `t("scoped.key")`** - never use `ErrorResponseTypes.*.errorKey` as a message.

**Internal module** (only called from its own `route.ts`) - receive `t` directly:

```typescript
import type { scopedTranslation } from "./i18n"; // type-only import

type ModuleT = ReturnType<typeof scopedTranslation.scopedT>["t"];

export async function createSmtpAccount(params: {
  data: CreateSmtpAccountInput;
  logger: EndpointLogger;
  t: ModuleT; // ← receive t directly, not locale
}): Promise<ResponseType<SmtpAccount>> {
  const { data, logger, t } = params;
  try {
    // ... db work ...
    return success({
      /* ... */
    });
  } catch (error) {
    logger.error(t("create.errors.failed"), parseError(error));
    return fail({
      message: t("create.errors.server.title"), // ← t(), NOT ErrorResponseTypes.errorKey
      errorType: ErrorResponseTypes.INTERNAL_ERROR,
    });
  }
}
```

**External module** (called from native repos, ai-stream, cron, story pages, etc.) - accept `locale` and create `t` internally:

```typescript
import { scopedTranslation } from "./i18n"; // value import (not type-only)
import type { CountryLanguage } from "@/i18n/core/config";

export class ThreadByIdRepository {
  static async getThreadById(
    threadId: string,
    user: JwtPayloadType,
    locale: CountryLanguage, // ← accept locale when called from outside
    logger: EndpointLogger,
  ): Promise<ResponseType<ThreadOutput>> {
    const { t } = scopedTranslation.scopedT(locale); // ✅ create t internally
    try {
      // ... db work ...
    } catch (error) {
      return fail({
        message: t("get.errors.server.title"),
        errorType: ErrorResponseTypes.INTERNAL_ERROR,
      });
    }
  }
}
```

### route.ts

**Internal module**: Pass `t` from handler props to the repository - never pass `locale`:

```typescript
handler: async ({ data, logger, t }) => {
  return SmtpRepository.createSmtpAccount({ data, logger, t });
},
```

**External module**: Pass `locale` from handler props to the repository:

```typescript
handler: async ({ urlPathParams, user, locale, logger }) =>
  ThreadByIdRepository.getThreadById(urlPathParams.threadId, user, locale, logger),
```

For inline handlers without a repository:

```typescript
handler: async ({ t, logger }) => {
  try {
    // ...
    return success({ /* ... */ });
  } catch (error) {
    return fail({ message: t("errors.server.title"), errorType: ErrorResponseTypes.INTERNAL_ERROR });
  }
},
```

## Common Mistakes

```typescript
// ❌ Hardcoded or default locale - ALWAYS FORBIDDEN
const { t } = scopedTranslation.scopedT("en-GLOBAL");   // WORST: hardcoded locale
const { t } = scopedTranslation.scopedT(defaultLocale); // FORBIDDEN: default locale shortcut

// ❌ Route creating t itself instead of using handler prop - ALWAYS FORBIDDEN
handler: async (props) => {
  const { t } = scopedTranslation.scopedT(props.locale); // ← WRONG: t already in props!
  return myRepo(data, t);
}

// ❌ Internal module accepting locale when it's only called from its own route.ts
// (prefer the cleaner t pattern for internal-only modules)
async function myInternalRepo(data, locale: CountryLanguage) {
  const { t } = scopedTranslation.scopedT(locale); // ← unnecessarily complex
}
// ✅ Internal module accepting t directly from route handler props
async function myInternalRepo(params: { data: MyInput; logger: EndpointLogger; t: ModuleT }) { ... }

// ❌ External module accepting t - won't work from native/cron/stream callers
static async getThreadById(threadId: string, user: JwtPayloadType, t: ModuleT, logger: EndpointLogger)
// ✅ External module accepting locale and creating t internally
static async getThreadById(threadId: string, user: JwtPayloadType, locale: CountryLanguage, logger: EndpointLogger) {
  const { t } = scopedTranslation.scopedT(locale);
}

// ❌ definition importing parent i18n
import { scopedTranslation } from "../../i18n"; // wrong scope
// ✅ definition importing module-local i18n
import { scopedTranslation } from "./i18n"; // same scope as repository

// ❌ Old global key format
t("app.api.emails.smtpClient.create.title");
// ✅ Scoped key
t("create.title");

// ❌ Missing scopedTranslation on createEnumOptions
createEnumOptions({ ACTIVE: "enums.status.active" });
// ✅ With scopedTranslation for compile-time key validation
createEnumOptions(scopedTranslation, { ACTIVE: "enums.status.active" });

// ❌ ErrorResponseTypes shortcut in fail()
fail({ message: ErrorResponseTypes.INTERNAL_ERROR.errorKey, ... });
// ✅ Scoped t() in fail()
fail({ message: t("errors.server.title"), ... });

// ❌ as const on translations
export const translations = { ... } as const;
// ✅ No as const
export const translations = { ... };

// ❌ No type annotation in German/Polish
export const translations = { create: { title: "Erstelle" } };
// ✅ Typed from English
import type { translations as enTranslations } from "../en";
export const translations: typeof enTranslations = { create: { title: "Erstelle" } };
```

## Migration Steps

1. **Create i18n structure**: `mkdir -p i18n/{en,de,pl} && touch i18n/index.ts i18n/en/index.ts i18n/de/index.ts i18n/pl/index.ts`
2. **Write English translations** (en/index.ts) with logical domains: `create`, `list`, `enums`, `errors`
3. **Translate** de/pl with `: typeof enTranslations` annotation
4. **Create i18n/index.ts** factory with `createScopedTranslation()` and export `ModuleTranslationKey`
5. **Update definition.ts**: add `scopedTranslation` import + pass to `createEndpoint()`, replace old keys
6. **Update enum.ts**: add `<ModuleTranslationKey>` generic, replace old keys with scoped keys
7. **Update repository.ts**: determine if module is internal-only or external-facing:
   - **Internal** (only called from its own `route.ts`): add `t: ModuleT` param, type-only import scopedTranslation
   - **External** (called from native repos, ai-stream, cron, etc.): add `locale: CountryLanguage` param, value import scopedTranslation, create `t` internally
   - Replace all `message:` strings with `t("...")`
8. **Update route.ts**: pass `t` (internal) or `locale` (external) from handler props to repository calls

### Database Enum Values

If enum values are stored in DB columns, check before migrating:

```bash
vibe sql "SELECT DISTINCT status FROM your_table LIMIT 20"
```

If old `"app.api.*"` values are stored, create a Drizzle migration to update them.

## Quick Reference

| File               | Pattern                                                                                                                                                                                                                          |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `i18n/index.ts`    | `createScopedTranslation({en, de, pl})` + `export type ModuleTranslationKey`                                                                                                                                                     |
| `i18n/en/index.ts` | `export const translations = { ... }` (no type, no as const)                                                                                                                                                                     |
| `i18n/de/index.ts` | `import type { translations as enTranslations } from "../en"` + `: typeof enTranslations`                                                                                                                                        |
| `definition.ts`    | `import { scopedTranslation } from "./i18n"` → pass to `createEndpoint()` + use `scoped*` field functions                                                                                                                        |
| `enum.ts`          | `createEnumOptions(scopedTranslation, { KEY: "enums.name.key" })` - value-import scopedTranslation                                                                                                                               |
| `repository.ts`    | **Internal**: accept `t: ReturnType<typeof scopedTranslation.scopedT>["t"]` (type-only import). **External**: accept `locale: CountryLanguage`, value-import scopedTranslation, create `t` internally. Use `t()` in all `fail()` |
| `route.ts`         | **Internal**: pass `t` from handler props. **External**: pass `locale` from handler props                                                                                                                                        |
